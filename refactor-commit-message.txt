refactor: apply hexagonal architecture to domain service implementations

Separate domain service interfaces from implementations following
hexagonal architecture principles:

Domain Layer (Interfaces Only):
- CategoryResolutionService: Interface for type-to-category resolution
- SubscriptionValidator: Interface for validation business rules
- LegacyDataMigrator: Interface for data migration logic
- ValidationResult: Sealed class for validation results
- MigrationException: Domain exception

Application Layer (Implementations):
- DefaultCategoryResolutionService: Concrete implementation
- DefaultSubscriptionValidator: Concrete validator
- DefaultLegacyDataMigrator: Concrete migrator
- All implementations moved from domain to application layer

Configuration Updates:
- UseCaseConfiguration: Updated imports to application layer
- Test files: Updated to import implementations from application layer

JaCoCo Coverage Rules:
- Exclude domain.service (interfaces) from coverage requirements
- Focus coverage on domain.model and application implementations
- Maintain 80% line coverage for implementation packages

Benefits:
- Clear separation of concerns (ports vs adapters)
- Domain layer is now pure (no infrastructure dependencies)
- Implementations properly located in application layer
- Better testability and maintainability
- Follows hexagonal architecture pattern correctly

All tests passing with 80%+ coverage maintained.
